# Dataset Entry Generator

## Philosophy

Dataset quality determines evaluation quality. Synthetic, lorem-ipsum test data produces meaningless evaluation results. This skill transforms abstract combination matrices into concrete, realistic test entries that mirror production data.

Each entry must be:
- **Domain-authentic**: Reads like real user input from the target domain
- **Scenario-specific**: Precisely matches its combination's dimensional constraints
- **Bidirectional**: Contains both input (what the prompt receives) AND expected output (what it should produce)
- **Evaluation-ready**: Structured for automated parsing, comparison, and scoring

This is Skill 6 in the prompt engineering workflow, following dataset combination generation.

## Workflow

### 1. Load Combinations

Read the combinations file generated by Skill 5:

```bash
cat projects/<name>/prompts/<prompt-name>/dataset/combinations.json
```

Validate structure:
- Each combination has: id, category, dimensions, description
- Categories align with mindmap structure
- Dimensions are clearly defined

### 2. Load Sample Data

Read sample data files from the grounding directory:

```bash
ls projects/<name>/grounding/examples/
```

Expected files:
- `sample_inputs.json` — Real input examples from the domain
- `sample_outputs.json` — Real output examples from the domain
- `sample_pairs.json` — Input-output pairs (ideal format)

Any of these may exist. Read whatever is available.

**If no sample data files exist:**
Ask the user to provide them:
> "I need sample data files in `grounding/examples/`. Please provide real (or anonymized) input/output examples from your domain as JSON files. These will be used to ground the generated test entries in realistic data patterns."

**Critical**: Do NOT proceed with generic/placeholder data. The quality of generated entries depends entirely on realistic samples. If the user provides data inline, save it to `grounding/examples/` before continuing.

### 3. Define Format Rules

Ask the user:
> "What format should each dataset entry follow?"

Collect:
- **Input schema**: Field names, data types, required vs optional fields
- **Output schema**: Expected structure of the prompt's response
- **Constraints**: Character limits, enum values, validation rules
- **Metadata requirements**: What context should be captured per entry?

Default format if not specified:
```json
{
  "id": "comb-001",
  "category": "category-name",
  "input": {
    "field1": "value1",
    "field2": "value2"
  },
  "expected_output": {
    "result": "expected result"
  },
  "metadata": {
    "dimensions": {},
    "notes": ""
  }
}
```

### 4. Load Grounding Documents

Check for domain context:

```bash
ls projects/<name>/grounding/
```

Read domain docs, glossaries, and reference materials to ensure entries use correct terminology, realistic field values, and domain-appropriate scenarios.

### 5. Generate Entries

For each combination:

1. **Parse dimensions**: Understand what this scenario requires
2. **Draft input**: Create realistic input data that satisfies all dimensional constraints
3. **Draft expected output**: Write the correct/ideal output the prompt should produce
4. **Add metadata**: Link back to combination dimensions, add explanatory notes
5. **Validate**: Check against format rules and domain realism

**Generation principles**:
- Use realistic names (diverse, culturally appropriate, not "John Doe")
- Use realistic numbers (actual price ranges, reasonable dates, valid IDs)
- Use domain vocabulary from grounding docs
- Vary data patterns (don't template-fill, make each genuinely different)
- Include edge cases naturally (empty fields if valid, boundary values)
- Make expected outputs specific and verifiable

### 6. Validate Entries

For each generated entry:
- Confirm input matches combination dimensions exactly
- Verify expected output is deterministic and clear
- Check JSON is valid and parseable
- Ensure no placeholder text remains ("TODO", "example", "lorem ipsum")
- Validate against user-provided format rules

### 7. Write Output

Write entries to dataset file:

```bash
# Output location
projects/<name>/prompts/<prompt-name>/dataset/entries.json
```

Structure:
```json
{
  "metadata": {
    "generated_at": "2026-05-13T10:30:00Z",
    "total_entries": 20,
    "source_combinations": "combinations.json",
    "format_version": "1.0"
  },
  "entries": [
    {
      "id": "comb-001",
      "category": "basic-queries",
      "input": { ... },
      "expected_output": { ... },
      "metadata": {
        "dimensions": {
          "complexity": "simple",
          "user_type": "novice"
        },
        "notes": "Testing basic user with straightforward query"
      }
    },
    ...
  ]
}
```

### 8. Output Summary

Report to user:
- Number of entries generated
- Breakdown by category
- Sample entry for review
- File location
- Next steps: "Dataset ready for evaluation (Skill 7: prompt-evaluator)"

## Principles

1. **Realism Over Convenience**: Never use placeholder data. Every field must contain domain-appropriate, realistic values.

2. **Scenario Fidelity**: Each entry must precisely match its source combination's dimensional constraints. If a combination specifies "novice user + complex query", the entry must reflect that exact scenario.

3. **Bidirectional Completeness**: Both input and expected output are required. Evaluators need ground truth to score against.

4. **Evaluability**: Outputs must be deterministic and verifiable. Avoid subjective expected outputs like "good response" — use specific, measurable criteria.

5. **Diversity Within Constraints**: Even entries in the same category should vary in surface details (different names, numbers, phrasing) while maintaining categorical consistency.

6. **Production-Ready Format**: Entries should be structured for immediate use in automated evaluation pipelines — valid JSON, consistent schema, no manual cleanup required.

7. **Domain Grounding**: Use terminology, patterns, and values from the grounding documents. If the domain is healthcare, use medical terminology; if finance, use financial instruments and regulations.

8. **Edge Case Authenticity**: Edge cases should feel real, not contrived. They should represent actual scenarios that might occur in production, not artificial stress tests.

9. **Metadata Richness**: Capture enough metadata to understand WHY an entry exists and WHAT it tests. Future readers should grasp the test's purpose from metadata alone.

10. **Validation Before Writing**: Every entry must pass format validation before being written to entries.json. No exceptions.

## Integration with Workflow

**Input from**: Skill 5 (combination generator) → `combinations.json`

**Output to**: Skill 7 (prompt evaluator) → `entries.json`

**Required context**:
- Domain grounding docs (for realistic data)
- Sample data files in `grounding/examples/` (for format reference and data patterns)
- Format rules (for validation)

**Success criteria**:
- All combinations have corresponding entries
- All entries pass format validation
- No placeholder/lorem ipsum data
- Expected outputs are clear and verifiable
- File is immediately usable for evaluation

## Example Interaction

**User**: "Generate dataset entries from combinations"

**Skill**:
1. Reads `projects/customer-analysis/prompts/search-query/dataset/combinations.json`
2. Reads `projects/customer-analysis/grounding/examples/sample_pairs.json` (real search queries + results)
3. Reads `projects/customer-analysis/grounding/domain_context.md` (customer data schema, valid regions, tier names)
4. Asks: "What format constraints should each dataset entry follow?"
5. User specifies: input query + filters, expected results with customer IDs
7. Generates 30 entries (one per combination):
   - Each has realistic customer data
   - Each matches its combination dimensions
   - Each includes expected results with specific customer IDs
8. Validates: all JSON valid, no placeholders, schema compliance
9. Writes to `entries.json`
10. Reports: "Generated 30 entries across 5 categories. Sample entry: [shows one]. Ready for evaluation."

## Notes

- If combinations.json doesn't exist, guide user to run Skill 5 first
- If sample data is too generic, push back and request domain-specific examples
- If expected outputs are ambiguous, ask user to clarify success criteria
- Store any generation rules/templates in project docs for reproducibility
- Consider versioning entries.json if regenerating (entries-v2.json, etc.)
